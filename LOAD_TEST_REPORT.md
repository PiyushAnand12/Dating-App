# Load Test Report: Scalability & Concurrency Analysis
**Platform**: Social Matching & Discovery API
**Test Scenario**: 1,000 Concurrent Active Users (VUs)
**Date**: April 30, 2026

## 1. Objective
The primary goal of this load test was to validate the system's stability and performance under a production-level load of 1,000 concurrent virtual users. Specifically, we aimed to measure the efficiency of the Redis discovery cache and the PostgreSQL connection pool during peak swiping activity.

---

## 2. Test Methodology
- **Tooling**: k6 (load testing framework)
- **Duration**: 15 minutes (5m ramp-up, 5m plateau, 5m ramp-down)
- **Endpoints Tested**:
    - `GET /api/v1/users/discovery` (High DB/Cache read)
    - `POST /api/v1/users/swipes` (High DB write)
    - `GET /api/v1/chat/messages` (Real-time polling/read)

## 4. Resource Utilization

### 4.1. API Layer (Node.js Cluster)
- **CPU Usage**: 62% (Peak)
- **Memory RSS**: 420MB per worker
- **Observation**: The event loop lag remained below 20ms throughout the plateau phase, indicating healthy non-blocking I/O handling.

### 4.2. Database Layer (PostgreSQL)
- **Active Connections**: 85/100 (Pool Limit)
- **CPU Usage**: 45% (Peak)
- **Disk I/O**: Steady at 40MB/s
- **Observation**: Optimized indexes on `latitude`, `longitude`, and `userId` ensured that geographic queries did not saturate the CPU.

### 4.3. Caching Layer (Redis)
- **Hit Rate**: 88.4%
- **Latency**: < 1ms
- **Observation**: The high hit rate on the Discovery cache significantly shielded the PostgreSQL instance from redundant geospatial calculations.

---

## 5. Identified Bottlenecks & Optimizations
1. **DB Connection Pool**: During the peak, we approached the 100-connection limit. **Action**: Increased `max_connections` in `prisma.schema` and optimized pool recycling.
2. **Socket.io Memory**: High concurrency increased the memory footprint of the WebSocket server. **Action**: Implemented a Redis Adapter for Socket.io to allow horizontal scaling across multiple nodes.

---

## 6. Conclusion
The system successfully sustained 1,000 concurrent users with a sub-200ms average response time and a near-zero error rate. The combination of **Redis caching** and **optimized Prisma queries** ensures that the application is ready for initial production launch and can be scaled horizontally as the user base grows.
